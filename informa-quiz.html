Informa Quiz App

By Callum Morrison

11-Feb-2020

# Informa Quiz App - About

Background & Brief

In the winter of 2018 / 19 I was tasked by Informa Systems to build a quiz app. I was the sole developer and the project was from scratch.

Informa System's main software is a feature-heavy LMS targeted at U.S. emergency services, mostly Police. As such, it was decided that a good way to market the main software was to create a completely free multiplayer quiz app. It was envisioned that such an app would be used as a training tool within police academies and the like. The brief was as follows:

  • Multiplayer quiz with single quiz master - multiple quiz player relationship.
  • Target platforms (to work together): Android, iOS and Chromebook.
  • Ability to select pre-made quizzes and to create quizzes.
  • Bonus: Single player mode.

Completion

The prototype was created in full and worked very well, please see the videos below. The app was briefly on the Play Store so Informa Systems customers could test. The brief was completed in full.

The project was unfortunately pulled before going into production due to changing business focus.

# Gameplay

General

This app was a text-based quiz game. Simply read the question and select one of the multiple choice answers:

Multiplayer

The main focus of the game was the asymetric multiplayer. One player is the "Question Master" - they choose the questions, settings, and create a session. The QM then starts the game and dictates the flow: this video shows the options available to the QM:

All of the other participants are "players". Players join the session via a code the QM would verbally share. The following video shows how a player can join a session using a temporary account and then answer questions (note that only 2 questions are shown and then the QM prematurely ends the game):

Single Player

Another mode included was Single Player. The player would choose a question-set and aim to finish this as fast as possible. If an incorrect answer was selected it would add a time penality and the player would have to choose again until they selected the correct answer. Their time was saved to a leaderbaord and the player could share a picture of their score via native options.

Question Creator

This featured allowed for the user to create a question-set and save it locally. If time had allowed, this feature would have been extended to allow question-sets to be shared and eventually some sort of platform.

# Technologies

The app was written in C#, built using Unity and the Unity Google Firebase integration.

Unity: This was chosen as the front-end / application framework for prototyping. Although Unity is a heavyweight choice, at the time it was the framework I was the most knowledgeable with as I had just developed a 3D game with it: Cubefall. It was our intention that the project would be recreated using something more suitable post product validation.

Firebase: This was used for the back-end, specifically authentication and database. Firebase has an extremely responsive real-time database, which proved perfect for our multiplayer quiz app where players receiving the question at the same time is key. A further bonus was that this setup meant our quiz sessions had a practically infinite upper player limit as literally thousands can interact with this relational database concurrently.

# Lessons

Here is a collection of thoughts and lessons learned, written with over a year to reflect.

Even though the brief was completed in full and all deadlines were met, the app was still cancelled. I think there is a lesson to be learned from the story of how the app started and how it ended.

The app started as a marketing tool for the businesses’ LMS software InformaOne. Our target user was an instructor at a police or firefighter academy. They would use our software as a fun tool to liven up dry classes, and in return this would generate interest in our main software.

During development, scope crept. We used the app “Kahoot” as a reference as it is a best-of-class example for multiplayer quiz apps. Kahoot is a fantastically successful company, valued north of $300m. This starts the inevitable thoughts: “if our base app does the same thing, why not try take a slice of the same pie?”. We slowly transitioned from a marketing tool into a Kahoot competitor.

The need for a question-set creator was there from the start, but now it we would need some kind of sharing platform, which would require some kind of curation system, which requires some kind of rating system. Some kind is the indicator of scope creep.

Alongside these new features, the app, which was still technically a prototype, was far off the polished UI and UX of Kahoot. This was ultimately why it was cancelled as the business decided it would be too costly to compete with the much larger, much better funded Kahoot studio.

The app was cancelled for a reason that shouldn’t have applied to it. I can’t criticise management in this decision as I was equally complicit: the thought of being in at the ground floor of a potentially multi-million dollar app was just too inviting. Management and I bounced ideas off each other and said yes without properly thinking through what yes really meant. It’s easy to say “yes” today, but hard to implement “yes” tomorrow, and even hard to make a project successful when twenty “yesses” has cornered your product.

I have thought of many questions now that would have been useful to ask myself back then, but for the sake of memorability I think of just this one: What is the intention of this new pivot in relation to the original concept? Just answering this one question helps you retrace how you got from 0 to where you are now, which grounds you regarding your next potential step.
